Transformations — what we do and why
Operation
What it does
Pandas / numpy call
Surrogate key
Replaces natural PK with a DW-standard integer key. Natural PKs (vehicle_id) can gap, reset, or collide across systems. Surrogate keys are stable and independent of source.
range(last_max+1, n)
String composition
Merges model_code + variant into model_variant_name. The target DW wants a single descriptive label per vehicle for report grouping, not two separate dimension lookups.
col_a + "_" + col_b
Date decomposition
Splits production_date into production_year and production_month. Analysts query "all vehicles from Q3 2022" constantly — pre-extracting avoids repeated YEAR() calls in every query.
.dt.year / .dt.month
Multi-condition band
Converts quality_score (float) into quality_tier (string bucket). Business users don't think in raw scores — they think in tiers. np.select handles 3+ conditions cleanly without chained np.where.
np.select(conds, vals)
Binary condition
Flags weight_category (HEAVY/LIGHT) and is_electric_vehicle (1/0). np.where is faster than .apply() for single yes/no conditions — it operates on the whole array at C speed.
np.where(cond, a, b)
Numeric multiply
Computes total_cost_eur = quantity × unit_cost per part row. The source records unit price only — the DW needs the line total for cost aggregations without forcing every analyst to recalculate it.
df[a] * df[b]
Unit conversion
Converts install_time_min to installation_hrs. Engineers log in minutes; finance and scheduling reports use hours. Standardising the unit in the DW removes the conversion burden from every consumer.
col / 60).round(2)
Null-to-numeric
Derives has_defect from whether defect_code is null. Null-checking in SQL is verbose and error-prone. A pre-computed 0/1 flag makes WHERE has_defect=1 trivial and allows SUM() aggregations.
.notna().astype(int)
String-to-numeric
Converts pass_fail ("PASS"/"FAIL") to is_passed (1/0). String columns can't be summed or averaged. Numeric flags allow pass_rate = AVG(is_passed) and COUNT(is_passed) directly in SQL.
(col=="PASS").astype(int)
Cost derivation
Computes rework_cost_usd = rework_hours × $85. Baking in the labour rate assumption makes the calculation auditable and versioned — if the rate changes, one config value changes, not every report.
.fillna(0) * 85
Dict map
Maps tier (1/2/3) to tier_label (STRATEGIC/PREFERRED/APPROVED). Dictionary maps are cleaner than np.select for small static lookups and produce readable labels for dashboards.
.map({1:"X", 2:"Y"})
Timedelta days
Computes contract_duration_days from two date columns. Lets procurement analysts filter "contracts expiring in under 90 days" without computing dates in every query.
(end - start).dt.days
Column rename
Applies the entire source→target name mapping in one step using the COLUMN_MAPPINGS dict. A single .rename() call is the cleanest way to apply all name changes before .to_sql().
.rename(columns=map)
Column selection
Drops source columns not in the target schema (variant, created_at, line_number, currency, install_time_min, is_active, pass_fail). Subset selection before load prevents schema errors.
df[list(mapping.values())]
Assertions — what we check and why it matters
Table
Type
What is being checked and why
vehicles
null check
VIN must never be null — it is the legal identity of a vehicle globally. A null VIN means the record is untraceable.
vehicles
format check
VIN is exactly 17 characters by international standard (ISO 3779). Shorter or longer means the source system truncated or corrupted it.
vehicles
uniqueness
No two vehicles can share a VIN. A duplicate indicates either a data entry error or a join that inadvertently created duplicate rows.
vehicles
range check
quality_score must be 0–100. Values outside this range are physically impossible and likely indicate a scoring formula bug upstream.
vehicles
enum check
shift must be MORNING / AFTERNOON / NIGHT. Any other value means the source system accepted a free-text entry that will break downstream grouping.
vehicles
enum check
status must be one of 4 valid states. Unknown states would create a 5th bucket in every status report — silently skewing completion metrics.
vehicles
range check
weight_kg must be positive. A zero or negative weight is physically impossible and would corrupt weight-based aggregations.
vehicles
null check
production_date must be present. Null dates break all time-series partitioning and make it impossible to assign the vehicle to a production period.
parts
null check
part_number must be present — it is the catalogue reference used for procurement, recall tracking, and warranty claims.
parts
range check
unit_cost must be positive. A zero or negative cost is likely a data entry error and would distort all cost-of-goods analytics.
parts
range check
quantity must be ≥ 1. Zero-quantity BOM lines are meaningless records that inflate part counts without contributing to cost.
parts
enum check
currency must be a recognised code. Unknown currencies would corrupt the total_cost derivation and make financial rollups unreliable.
parts
enum check
defect_flag must be 0 or 1. Any other value means the flag column is being misused, potentially masking real defect data.
parts
referential integrity
Every vehicle_id in parts must exist in vehicles. Orphaned part records would be invisible in any vehicle-joined report — silent data loss.
suppliers
uniqueness
supplier_code must be unique. Duplicates mean two rows claim to be the same supplier, causing double-counting in cost and parts-by-supplier reports.
suppliers
range check
rating must be 0–5. Out-of-range values mean the scoring system was misconfigured and ratings cannot be compared meaningfully.
suppliers
enum check
tier must be 1, 2, or 3. Any other value breaks the tier_label mapping and places suppliers outside the supply-chain hierarchy.
suppliers
enum check
is_active must be 0 or 1. Unexpected values mean the active flag is inconsistently populated and active_status derivation will produce wrong results.
suppliers
date logic
contract_start must be before contract_end. Reversed dates mean the contract duration would be negative — a clear data quality failure in the source system.
quality_checks
enum check
pass_fail must be exactly "PASS" or "FAIL". Any other value breaks the is_passed derivation and corrupts the pass-rate metric across the entire QC dataset.
quality_checks
range check
rework_hours must be ≥ 0. Negative rework hours are physically impossible and would produce negative rework_cost_usd values.
quality_checks
referential integrity
Every vehicle_id in quality_checks must exist in vehicles. Orphaned QC records cannot be attributed to a vehicle — they become unanalysable noise.
quality_checks
enum check
test_type must be one of the 5 defined test categories. Unknown types would create spurious categories in defect-rate-by-test-type analysis.
